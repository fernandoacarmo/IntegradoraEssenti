Objetivo

Este documento visa especificar os métodos das web services e os parâmetros de comunicação necessários para a integração entre o Sistema Ajuri e o Sistema de WMS.

Padrão de Comunicação

A integração do Sistema de WMS com o Sistema Ajuri é realizada através de Web Services.

Os serviços são acessíveis através do protocolo SOAP (Simple Object Access Protocol). Este protocolo trata-se de uma especificação que possibilita a troca de informações de uma forma estruturada. As mensagens trocadas entre o cliente e o serviço adotam o padrão de formato de mensagens em XML (Extensible Markup Language).

A interface dos serviços pode ser consultada pela URL do arquivo WSDL(Web Service Description Language), um documento no padrão XML que descreve o funcionamento de um Web Service.

Tanto o Ajuri quanto o Sistema de WMS deverão disponibilizar um conjunto de web métodos presentes em seu Web Service com as especificações técnicas detalhadas nesta documentação.

Vale ressaltar também, que todos os arquivos XML especificados neste documento serão validados por um arquivo XML Schema, responsável pela definição de todos os campos, atributos, tipos e valores aceitos nas tags.

  • Método
    • ATUALIZAR_SAIDA

Este método tem como objetivo receber os dados de um Pedido de Saída expedido no Sistema de WMS, bem como seus itens, e executar as baixas no estoque do Sistema Ajuri.

Será necessário informar o Ajuri a real quantidade que foi expedida pelo WMS, já que em determinadas situações de operação o WMS poderá expedir uma quantidade menor que a requerida pelo pedido de saída. Por exemplo: o pedido de saída necessita de 50 qtde. de X produto, mas em estoque, o WMS tem apenas 25 qtde do produto X. Porem o gestor do armazém em conjunto com a instituição que necessitam do produto, decidem expedir somente as 25 qtde mesmo, “cortando” o pedido de saída, logo, o Ajuri necessita saber dessa informação tendo o retorno da expedição do WMS.

Especificação das Web Services do Sistema de WMS

Todos os dados trafegados devem estar no padrão XML e ser validados por um arquivo XML Schema contendo todas as definições, chamado de XSD (XML Schema Definition), que descreve a estrutura de um documento XML.

Elementos comuns

Ticket

Em todos os arquivos XML trafegados existe um conjunto de tags para identificar o ticket, ou seja, o sistema e identificador que geraram aquele arquivo.

Abaixo segue a especificação de como deverá preenchido quando for informado a partir do Ajuri:

<vx:ticket>
  <vx:pid>AJURI</vx:pid>
  <vx:nome>AJURI</vx:nome>
  <vx:empresa>RV</vx:empresa>
  <!-- empresa fornecedora do aplicativo que gerou o arquivo -->
  <vx:fornecedor>AJURI</vx:fornecedor>
  <!-- coloque o IP do host que gerou o arquivo -->
  <vx:host>172.16.0.0</vx:host> 
  <!-- sempre siga este padrão de data e hora com o “T” entre elas -->
  <vx:data-origem>2010-07-27T12:00:00</vx:data-origem> 
  <!-- coloque a versão do aplicativo que gerou o arquivo -->
  <vx:versao>1.0.0</vx:versao>
  <!-- coloque alguma observação pertinente -->
  <vx:observacoes></vx:observacoes>
  <!-- identificador único do ticket -->
  <vx:id></vx:id>
</vx:ticket>
Gerador

A tag gerador encontrada em alguns dos arquivos xml deve levar dados sobre o usuário que acionou a criação do arquivo. Abaixo segue um fragmento que deverá ser gerado pelo Sistema Ajuri.

<vx:gerador>
  <!-- CHUCK é um valor fictício, substitua-o pelo login do usuário -->
    <vx:login>AJURI</vx:login>
  <!-- JONH CHUCK é um valor fictício, substitua-o pelo nome do usuário -->
    <vx:nome>AJURI</vx:nome>
  <!-- sempre coloque GERADOR -->
    <vx:atribuicao>GERADOR</vx:atribuicao>
  <!-- coloque o IP do host do usuário -->
    <vx:host>172.16.0.2</vx:host>
  <!-- sempre siga este padrão de data e hora com o “T” entre elas -->
    <vx:data>2010-07-29T12:00:00</vx:data>
  <!-- sempre coloque EXPORTAR_DADOS -->
    <vx:operacao>EXPORTAR_DADOS</vx:operacao>
  <!-- sempre coloque AJURI -->
    <vx:pid>AJURI</vx:pid>
</vx:gerador>

Os Fornecedores, Fabricantes e Destinatários (Unidades de Saúde) serão cadastrados no Sistema Ajuri e repassados ao Sistema de WMS através do método integrarEntidadeMestra disponível na Web Service no endereço especificado acima.
Uma tentantiva de cadastro será feita e então o Sistema de WMS deverá retornar uma mensagem de sucesso ou falha na operação. Fluxo Básico:

Embalagem

No Sistema de WMS existe uma composição bastante complexa do material em função das necessidades de dados logísticos, porém no Sistema Ajuri praticamente só existem o código, descrição e a unidade de medida do material. Com isso diversos valores constantes serão aplicados, impactando diretamente em cálculos sobre cubagem, volumes, paletes e peso, ou seja, não tem-se valor real para cada um deles.

Devido a falta destes dados, eles deverão ser configurados valores padrões para cada registro de produto conforme o fragmento de xml abaixo:

<vx:configura-embalagens>
  <vx:configuracao>
    <vx:embalagem>
      <!-- sempre colocar o valor UNITARIA -->
      <vx:codigo>UNITARIA</vx:codigo>
      <!-- sempre colocar o valor 0.0001 -->
      <vx:comprimento-metros>0.0001</vx:comprimento-metros>
      <!-- sempre colocar o valor 0.0001 -->
      <vx:largura-metros>0.0001</vx:largura-metros>
      <!-- sempre colocar o valor 0.0001 -->
      <vx:altura-metros>0.0001</vx:altura-metros>
      <!-- sempre colocar o valor 0.0001 -->
      <vx:tara-kilos>0.0001</vx:tara-kilos>
    </vx:embalagem>
    <!-- sempre colocar o valor 1 -->
    <vx:itens-por-volume>1</vx:itens-por-volume>
    <!-- sempre colocar o valor 1 -->
    <vx:unidades-por-item>1</vx:unidades-por-item>
    <!-- sempre colocar o valor 1 -->
    <vx:volumes-por-palete>1</vx:volumes-por-palete>
    <!-- sempre colocar o valor 0.0001 -->
    <vx:peso-bruto-volume-kilos>0.0001</vx:peso-bruto-volume-kilos>
    <!-- sempre colocar o valor 0.0001 -->
    <vx:peso-liquido-volume-kilos>0.0001</vx:peso-liquido-volume-kilos>
  </vx:configuracao>
</vx:configura-embalagens>
